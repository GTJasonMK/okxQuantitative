# 阶段01：真实秒级采集启动链路设计

## 1. 目标
- 本阶段只解决“数据中心 -> 秒级采集”的真实运行闭环。
- 闭环定义：开始采集 -> 创建 `collection_session` -> 独立采集运行时启动 -> 订阅 OKX `trades + book_channel` -> 轮询 `mark/index/open_interest/funding/premium` -> 每秒 flush `second_state` -> 写入 `research_second_states` -> 刷新会话质量 -> 通过 `research_platform` 事件桥推送前端 -> 页面展示控制、进度、质量、历史。
- 本阶段不做：修改 [模型训练构思.txt](/mnt/e/Code/okxQuantitative/模型训练构思.txt)、重写 `target_census_15m`/dataset/training 主协议、跨会话拼接输入窗口、多合约并发采集、任何静默 fallback 或假采集结果。
- `执行计划2` 当前是冻结文档中的“采集子项目计划”，只覆盖 `collection_session + second_state + 采集事件/UI`。`boundary_target_15m`、`sample_index_15m`、独立 `target_census_15m`、dataset、training 仍属于后续阶段，不在本目录当前 5 个阶段内实现。

## 2. 冻结约束
- 采集单位：单合约单会话。
- 输入定义：数据集层只接受单会话内连续 `7200 x F` 的 `second_state` 输入窗口。
- 决策边界：交易所固定 `15m` 边界。
- anchor：`decision_ts = t` 时，`anchor = close_price(second_bucket = t - 1)`。
- 标签来源：必须由同一会话内后续 `900` 秒 `second_state` 自聚合，不依赖外部 `15m` K 线。
- 本阶段职责：只交付稳定 `second_state` 会话仓库和真实采集运行时。
- 版本纪律：完整性规则、字段协议、事件协议如有变动，必须升级版本号，禁止原地改语义。

## 3. 架构边界
- 新增独立域：`data_center_collection`。
- 负责：会话启停、OKX 实时订阅、1 秒条聚合、`research_second_states` 持久化、会话质量刷新、采集运行时事件推送。
- 不负责：dataset manifest、training run、trend research 页面业务、target census 主逻辑。
- 依赖方向冻结为：`data_center_collection runtime -> reusable market primitives -> research_platform storage / emit bridge`。
- `trend_research` 只允许输出可复用底层原语，不再承担本阶段实时采集运行时。

## 4. 组件拆分
### 4.1 `CollectionRuntimeRegistry`
- 维护 `session_id -> runner`。
- 防止同一活动会话重复启动。
- stop 时释放运行时句柄。
- 约束：同一时刻最多 1 个 `starting/running/stopping` 会话；未注册会话 stop 必须显式报错。

### 4.2 `CollectionSessionController`
- 接收 API start/stop 动作。
- 创建会话记录、驱动 runner 启停、回写会话状态与错误信息。
- 不负责 OKX 消息解析、秒聚合计算、前端渲染格式。

### 4.3 `MarketFeedAdapter`
- 订阅 `trades` 和 `book_channel`。
- 将 OKX 消息归一化为 `TradeTickEvent`、`BookTopEvent`。
- 可复用 `backend/app/core/websocket_manager.py` 的底层连接能力。
- 禁止在 adapter 中写库或维护 session 状态机。

### 4.4 `ContractStatePoller`
- 周期抓取 `mark_price/index_price/open_interest/funding_rate/premium`。
- 输出标准化 `ContractStateSnapshot`。
- 可复用 `build_contract_state_snapshot()` 原语。

### 4.5 `SecondBarAccumulator`
- 面向单合约当前秒桶累计 `trade/book/state`。
- 在秒边界 flush 成完整运行时快照。
- 负责秒内 `OHLC`、成交流 notional/count/burst、盘口 `mid/microprice/spread`、多档盘口结构、`oi_delta/funding_delta`、输入新鲜度。
- 硬约束：必须由秒边界时钟驱动 flush；不允许“有事件才写”；无效秒也要显式产出质量语义。

### 4.6 `SecondStateAssembler`
- 将 flush 结果映射为 `research_second_states` 协议行。
- 生成 `is_valid_second / quality_grade / invalid_reason`。
- 只有这一层知道 `SECOND_STATE_COLUMNS` 和 `strict_v1` 规则。

### 4.7 `CollectionEventPublisher`
- 向 `research_platform` 事件桥发运行时事件。
- 不推裸 OKX 消息，只推会话级或秒级业务事件。
- 最小事件集：`session_started`、`session_running`、`second_flushed`、`session_quality_updated`、`session_stopping`、`session_stopped`、`session_finished`、`session_failed`、兼容保留 `session_updated`。

## 5. 运行时数据流
1. 前端点击开始采集。
2. `CollectionSessionController.create_and_start()`。
3. 写入 `research_collection_sessions`。
4. `CollectionRuntimeRegistry` 注册 runner。
5. runner 并发启动 `MarketFeedAdapter`、`ContractStatePoller`、`SecondBoundaryClock`。
6. 所有输入进入 `SecondBarAccumulator`。
7. 每到一个整秒执行 `flush(second_bucket)`。
8. `SecondStateAssembler` 产出持久化 payload。
9. `SecondStateWriter` 用 `(session_id, second_bucket)` 幂等写库。
10. `SessionQualityTracker` 增量更新 coverage/stale 统计。
11. `CollectionEventPublisher` 推送事件到前端。
- 固定规则：任何一秒即使没有新成交，也必须产出 1 行 `second_state`；真正无效必须通过 `is_valid_second = 0` 暴露；`has_*_input` 只表示“本秒是否有新输入”，不能和最终字段值混义。

## 6. 会话状态机
- `research_collection_sessions.status` 扩展为：`starting`、`running`、`stopping`、`stopped`、`finished`、`failed`。
- 合法跃迁：`starting -> running`、`starting -> failed`、`running -> stopping`、`running -> finished`、`running -> failed`、`stopping -> stopped`、`stopping -> failed`。
- 禁止跃迁：`failed -> running`、`stopped -> running`、`finished -> running`。
- 启动成功定义：session 已落库；行情订阅建立成功；秒边界时钟启动成功；状态轮询至少完成一次初始化或显式记为缺失；至少 1 条 `research_second_states` 成功写入。只有满足这些条件，状态才允许由 `starting` 转为 `running`。
- 停止来源固定为：`manual_stop`、`planned_finish`、`process_shutdown`。
- session 层新增错误字段：`last_error_code`、`last_error_message`、`failed_at`。

## 7. `second_state` 字段协议
### 7.1 时间语义
- `second_bucket = s` 表示时间桶 `[s, s + 1)`。
- 输入窗口固定为 `{t - 7200, ..., t - 1}`。
- `anchor_close_price = close_price(second_bucket = t - 1)`。

### 7.2 字段组
- 标识与时间：`session_id, inst_id, second_bucket, ts_exchange, ts_local`。
- 快照类：`bid_price, ask_price, bid_size, ask_size, mid_price, microprice, mark_price, index_price, open_interest, funding_rate, premium`。
- 流量聚合类：`trade_count, signed_trade_notional, buy_notional, sell_notional, buy_count, sell_count, max_trade_notional, buy_burst_count, sell_burst_count, buy_burst_notional, sell_burst_notional, open_price, high_price, low_price, close_price, oi_delta, funding_delta`。
- 质量类：`has_trade_input, has_book_input, has_state_input, book_age_seconds, state_age_seconds, clock_skew_ms, is_valid_second, quality_grade, invalid_reason, integrity_policy_version`。

### 7.3 关键公式与回退规则
- `mid_price = (bid_price + ask_price) / 2`。
- `microprice = (bid_price * ask_size + ask_price * bid_size) / (bid_size + ask_size)`。
- `basis_bps = 1e4 * (mark_price - index_price) / index_price`。
- 若 `trade_count > 0`：`open_price` 为首笔成交价，`high_price` 为秒内最高成交价，`low_price` 为秒内最低成交价，`close_price` 为末笔成交价。
- 若 `trade_count = 0` 且盘口有效：`open_price = high_price = low_price = close_price = mid_price`。
- 若盘口也不可用：`open/high/low/close = 0`。
- book/state 快照允许沿用上一秒最后可见快照；流量聚合字段不允许跨秒累计；`has_book_input/has_state_input = 0` 时快照字段仍可非零，因为它们可能来自 carry-forward。

### 7.4 `strict_v1` 有效秒规则
- 固定阈值：`BOOK_WARN_STALE_SECONDS = 1.0`、`BOOK_INVALID_STALE_SECONDS = 2.0`、`STATE_WARN_STALE_SECONDS = 5.0`、`STATE_INVALID_STALE_SECONDS = 15.0`、`CLOCK_SKEW_WARN_MS = 1000`、`CLOCK_SKEW_INVALID_MS = 3000`。
- `is_valid_second = 1` 当且仅当：顶层盘口有效 `bid_price > 0`, `ask_price > 0`, `ask_price >= bid_price`；`mid_price > 0`, `microprice > 0`；状态快照有效 `mark_price > 0`, `index_price > 0`, `open_interest >= 0`；`book_age_seconds <= 2.0`；`state_age_seconds <= 15.0`；`clock_skew_ms <= 3000`；`open/high/low/close > 0`；并满足 `low_price <= min(open_price, close_price) <= max(open_price, close_price) <= high_price`。

### 7.5 `quality_grade` 与 `invalid_reason`
- `quality_grade = A`：有效，且 `book_age_seconds <= 1.0`、`state_age_seconds <= 5.0`、`clock_skew_ms <= 1000`。
- `quality_grade = B`：有效，但 freshness 落入 warning 区。
- `quality_grade = C`：有效，但当前秒没有新 `book/state` 输入，完全依赖 carry-forward。
- `quality_grade = F`：无效。
- `invalid_reason` 只允许固定 code，并用 `|` 连接：`missing_book_snapshot`、`missing_state_snapshot`、`book_stale`、`state_stale`、`invalid_top_of_book`、`invalid_price_path`、`clock_skew_exceeded`。

## 8. 前端 UI 协议
- 页面继续挂在数据中心 `秒级采集` 页，不新增独立过程诊断页。
- 固定四区块：
- 采集控制台：表单字段 `inst_id`, `planned_duration_sec`, `book_channel`, `sampling_policy_id`, `integrity_policy_version`, `collector_version`；动作 `开始采集/停止采集`；运行摘要 `status`, `session_id`, `started_at`, `elapsed`, `remaining`, `last_error_*`。
- 实时进度板：阶段条 `会话创建 -> 订阅建立 -> 首次状态快照 -> 首次秒条落库 -> 稳定运行`；计数 `written_seconds / planned_duration_sec`；`valid_second_count / invalid_second_count / missing_second_count`；距首个 `7200` 秒完整输入窗口剩余秒数；距下一个 `15m` 边界剩余秒数。
- 质量诊断区：输入源健康、新鲜度统计、`invalid_reason` 分布。
- 会话历史与回放：表格字段 `session_id`, `inst_id`, `status`, `started_at`, `elapsed`, `coverage_ratio`, `quality_score`, `valid_second_count`, `stop_reason`；详情区图表固定三类 `close_price`, `trade_count`, `spread_bps`。
- 状态呈现：`starting` 显示启动中且不用按钮闪烁表示状态；`running` 显示绿色状态牌和数值更新；`stopping` 禁用按钮并显示收尾中；`finished` 显示按计划完成；`stopped` 显示人工停止；`failed` 顶部错误横幅展示 `last_error_code / last_error_message`。

## 9. 事件协议
- 前端订阅频道保持为 `research_platform`。
- 事件集合：`session_started`、`session_running`、`session_stopping`、`session_stopped`、`session_finished`、`session_failed`、`second_flushed`、`session_quality_updated`、兼容事件 `session_updated`。
- 最小 payload 约束：所有 session 级事件必须带 `session_id`；`session_failed` 必须带 `status`, `error_code`, `error_message`, `failed_at`；`second_flushed` 必须带 `session_id`, `second_bucket`；`session_quality_updated` 必须带 coverage 摘要或能驱动前端重新拉取详情。
- 兼容策略：前期保留 `session_updated`；前端先做到“收到任意 session 事件就刷新详情”；之后再逐步消费细粒度事件做局部刷新。

## 10. 测试与验收
- 测试分四层：
- 后端运行时单元测试：`CollectionRuntimeRegistry`、`CollectionSessionController`、`SecondBarAccumulator`、`SecondStateAssembler`。
- 后端协议与存储测试：session 新字段 round-trip、`second_state` 字段 round-trip、`strict_v1` 有效秒判定、coverage/stale 统计。
- 事件流集成测试：fake market feed + fake clock + real storage；验证 `starting -> running -> stopped/finished/failed`；验证 `second_flushed` 与 `session_failed` 事件。
- 前端交互测试：workspace 事件响应、控制卡状态映射、历史会话表与详情联动、进度板与质量诊断区渲染。
- 验收条件必须全部满足：点击开始后 session 进入 `starting`；首条秒条成功写入后进入 `running`；每秒都能产出 1 行 `second_state`；无成交秒也有统一 `close_price`；关键输入 stale 超阈值时显式变成无效秒；点击停止后进入 `stopped`，自然到期进入 `finished`；关键错误进入 `failed` 且前端可见；页面同时完成控制、进度、质量、历史四类展示；自动化测试覆盖运行时、协议、事件、前端四层。

## 11. 本阶段输出
- 数据中心页面可启动采集。
- 后端存在独立采集运行时。
- `research_second_states` 会被真实持续写入。
- 会话状态、错误、质量能够被显式诊断。
- 前端不再只展示静态壳子，而是展示真实运行状态。
- dataset/training 层仍以上位冻结文档为准，但开始依赖本阶段稳定产出的 `second_state` 会话仓库。
